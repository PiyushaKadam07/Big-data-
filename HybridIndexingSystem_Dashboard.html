● The Problem
Why existing approaches fall short
Every large-scale system that stores data needs an index to find it quickly. But today's solutions force a tradeoff.
🏚️
Centralized Index
One server holds the entire index. Simple to manage, but crashes under load — every query competes for the same bottleneck.
✗ Single point of failure
✗ Latency grows with data
✗ Can't scale horizontally
📡
Distributed Index
Data is spread across many nodes. Scalable, but every query gets broadcast to ALL nodes — most of which have nothing relevant.
✓ Horizontally scalable
✗ Wastes network bandwidth
✗ High coordination cost
⬡
Hybrid (Proposed)
A metadata layer acts as a smart directory — routing each query only to the nodes that hold relevant data. Fast AND scalable.
✓ Metadata-driven routing
✓ Reduced communication
✓ Parallel + targeted

📦 Data Partitioning
Hash-based Partitioning
Each document is assigned to a node using consistent hashing on its ID. This ensures even distribution and minimal rebalancing when nodes are added.
Central Metadata Index
Stores a lightweight mapping of term → {node_ids} and doc → node. Fast O(1) lookups — not a bottleneck.
