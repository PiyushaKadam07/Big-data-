Benchmark 1 — Latency vs Dataset Size
Average Query Latency as Dataset Grows (4 nodes)
Shows how each approach scales as the number of indexed documents increases from 100 to 1,000.
Hybrid
Distributed
Centralized
Key finding: Hybrid and Distributed scale linearly with dataset size (O(n/k) per node), while Centralized degrades drastically — growing from 63ms at 100 docs to over 5,000ms at 1,000 docs due to sequential full-index scanning. Hybrid maintains a consistent ~5% latency advantage over pure distributed by avoiding unnecessary node communication overhead.
Benchmark 2 — Latency vs Node Count
Scalability: Average Query Latency as Cluster Grows (500 docs)
Shows how adding more nodes affects query latency. Centralized improves because the scan workload stays fixed; distributed and hybrid grow due to coordination overhead.

Distributed
Centralized (log scale)
Key finding: As node count grows, Distributed queries must broadcast to more nodes, increasing coordination cost. Hybrid's metadata-based routing is consistently 20–25% faster than pure Distributed at all node counts by pruning irrelevant nodes before dispatching queries.
Benchmark 3 — Query Routing Efficiency
Nodes Queried: Hybrid vs Distributed (per query type)
Demonstrates how the central metadata layer reduces the number of nodes queried per search.
Key finding: For specific queries (e.g. "NoSQL database", "Apache Spark"), the metadata layer routes to 1–2 nodes instead of broadcasting to all 4, reducing network communication by up to 75%. Broader queries touching multiple topic shards still benefit from parallel execution.
Benchmark 4 — Throughput Comparison
Queries per Second — All Three Modes
Effective throughput accounting for parallel execution capability and latency overhead.
Key finding: Hybrid achieves 62.2 q/s — over 2× the distributed baseline (29.6 q/s) and 300× the centralized approach (0.2 q/s at 1000 docs). The throughput advantage compounds with dataset size as centralized's latency degrades exponentially.
